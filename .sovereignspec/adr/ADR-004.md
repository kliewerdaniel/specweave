# ADR-004: A2A/MCP Gateway for Agent Federation

## Context
SpecWeave must interoperate with other local agents. Agents need to discover specs, request work via sub-spec delegation, submit completed work, and query impact analysis. Two protocols are relevant: A2A (Agent-to-Agent) for agent communication and MCP (Model Context Protocol) for tool integration.

## Decision
Implement a dual-protocol gateway:
- **A2A Protocol**: `GET /a2a/specs` (discover), `POST /a2a/delegate` (delegate sub-spec), `POST /a2a/submit` (submit result)
- **MCP Protocol**: `GET /mcp/tools` (list tools), `POST /mcp/execute` (execute tool)
- Both protocols share the same underlying SQLite persistence for delegation records
- All communications over localhost HTTP

## Rationale
- A2A provides agent-to-agent contract negotiation (discover → delegate → submit)
- MCP provides simpler tool-execution pattern for agents that just need single operations
- Shared persistence ensures delegation state survives restarts
- Both protocols are standard and tool-agnostic

## Alternatives Considered
- **Single custom protocol**: Simpler but not interoperable with existing agents
- **WebSocket for real-time**: Adds complexity; HTTP polling is sufficient for local-first
- **Message queue (RabbitMQ/NATS)**: Overkill for single-machine deployment

## Consequences
- Positive: Any A2A/MCP-compatible agent can interact with SpecWeave
- Positive: Delegation audit trail via SQLite
- Negative: Two protocols to maintain
- Negative: No real-time push (polling-based status checks)
