# ADR-002: Self-Verifying Compiler with 6 Sequential Gates

## Context
The compiler must verify spec correctness before generating tasks. Verification must be deterministic and auditable. The 12-step pipeline in the specweave.sspec design document describes 6 verification gates.

## Decision
Implement a sequential gate pipeline where gates run in order 1-6, stopping on the first failure:

1. **Gate 1 (Completeness)** — checks all required fields present
2. **Gate 2 (Consistency)** — checks no contradictions via symbolic validator
3. **Gate 3 (Dependencies)** — topological sort, cycle detection
4. **Gate 4 (Constraints)** — local-first, no-cloud, governance checks
5. **Gate 5 (Coherence)** — neural check of intent-architecture alignment
6. **Gate 6 (Readiness)** — modules/endpoints resolvable to code patterns

Each gate produces a `VerificationGate` record with `gate_id`, `status`, `failure_reason`, `failure_location`, and `checked_at`. Results are persisted to SQLite for audit.

## Rationale
- Sequential gates provide clear failure traceability
- Early gates (completeness) fail fast before expensive neural checks
- Persisted gates enable historical audit and trend analysis
- Failure locations guide targeted regeneration

## Alternatives Considered
- **Parallel gates**: Faster but lose traceability of failure chain
- **Single monolithic check**: Simpler but cannot pinpoint which aspect failed
- **Async gates with webhooks**: Over-engineered for local-first use

## Consequences
- Positive: Clear failure chain — fix first failure, re-run
- Positive: Each gate independently testable
- Negative: Neural gate (coherence) blocks subsequent gates if slow
- Mitigation: Configurable timeout per gate
